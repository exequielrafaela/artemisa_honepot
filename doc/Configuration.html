Configuration¶

The configuartion of Artemisa is performed through files with extension .conf situated in the /conf directory. In this section each file is detailed and it’s explained how to configure them.

artemisa.conf¶

In this file you will find important parameters related to the general operation of the program.

Environment section:

    [environment]
    local_ip=127.0.0.1
    local_port=5065
    sip_domain=127.0.0.1
    user_agent=MySIPphone/1.5
playfile=sample.wav
    behaviour_mode=active
    max_calls=2
  • local_ip: IP address of the machine (or an interface of it) where Artemisa runs
  • local_port: UDP port
  • sip_domain: network domain
  • user_agent: name that will appear in the User-Agent: field of the SIP header of the messages sent by Artemisa
  • playfile: name of the WAV file that will be played after sending an 200 OK to the caller. The file is searched into the /audiofiles directory. The format must be uncompressed 16 bit PCM or compressed G.711 A-law/U-law.
  • bahaviour_mode: behaviour mode of Artemisa. There are three possible modes: active, passive and aggressive. Each mode is defined on file behaviour.conf.
  • max_calls: maximum number of calls that Artemisa can answer at the same time

Sound section:

[sound]
enabled=true
device=0
rate=44100
  • enabled: if it contains ‘true’ the calls shall be recorded
  • device: index of the sound device to be used. DO NOT simply modify this line unless you are sure of what you are changing. In order to get a list of the available sound devices run Artemisa with command-line parameter ‘-g’.
  • rate: sample rate (may happen that your device doesn’t support certain rates)

E-mail section:

[email]
enabled=false
smtp_server_ip=smtp.somemail.com
smtp_server_port=587
smtp_server_use_tsl_ssl=true
smtp_server_username=artemisareports@yourcompany.com
smtp_server_password=thepassword
from_mail=artemisareports@yourcompany.com
recipients_mail=administrator@yourcompany.com, mike@yourcompany.com
to_header="Administrator"
subject="Artemisa report of activity"
  • enabled: if it’s set to ‘false’, no e-mail shall be sent.
  • smtp_server_ip: IP address of the SMPT server to use
  • smtp_server_port: SMPT server port address
  • smtp_server_use_tsl_ssl: use of encriptation
  • smtp_server_username: username
  • smtp_server_password: password
  • from_mail: From field address
  • recipients_mail: e-mail address of the recipients. You can add more than one recipient separated by comma.
  • to_header: To field address
  • subject: subject of the e-mail

extensions.conf¶

Here you must set up the extensions that shall be used by Artemisa in the registration process. The extensions must be valid.

In order to define an extension use the following format:

[extension]
username=
password=

For example, if you have a server and have decided to leave the extension 5000, 5001, 5002, 5003 and 5004 for Artemisa, the file should be configured like (supposing usernames and passwords):

[5000]
username="5000"
password=5000

[5001]
username="5001"
password=5001

[5002]
username="5002"
password=5002

[5003]
username="5003"
password=5003

[5004]
username="5004"
password=5004

Note

The extensions defined between square brackets must be unique.

servers.conf¶

Here you are able to set the registrar servers configuration that Artemisa shall use to register itself. Each server it’s configured inside a section with the following format:

[some name]
registrar_ip=127.0.0.1
registrar_port=5060
registrar_time=15
nat_keepalive_interval=0
exten=5000,5001,5002
  • registrar_ip: IP address of the SIP REGISTRAR
  • registrar_port: UDP port
  • registrar_time: time in minutes between automatic registrations
  • nat_keepalive_interval: when dealing with NAT proxies, you can set a value in seconds which indicates the time interval between keep alive messages. If zero is written, then the NAT keep alive messages shall not be sent.
  • exten: in this field you should set the extensions to be used. They must be declared in extensions.conf.

For example, you have a test SIP server running in the same maching of Artemisa. Furthermore, Artemisa registers itself to the real server of your company, which is found in a certain IP address. On the other hand, you want to use some extensions for the test server and some for the real server. Then, the configuration file should look like:

[local]
registrar_ip=127.0.0.1
registrar_port=5060
registrar_time=15
nat_keepalive_interval=0
exten=5003,5004

[myserver]
registrar_ip=192.168.0.205
registrar_port=5060
registrar_time=15
nat_keepalive_interval=0
exten=5000,5001,5002

behaviour.conf¶

Here each behavior mode is defined. There are just four sections, which names are [active], [passive], [aggressive] and [investigation].

Inside each section, the actions performed by Artemisa when a call is received are defined.

The list of available commands for the sections is defined in behaviour.conf itself.

The idea behind this, is that you can configure Artemisa in order to in some occasions just take the calls and answer them, which is a “passive” mode, and in other occasions to investigate the calls (“active” mode). You may want also that Artemisa counterattacks the intruder, which would be an “aggressive” mode.

Note

At the moment there are no actions available for an aggressive mode.

actions.conf¶

In this configuration file you can define the parameters used when calling the user-configurable scripts such as “on_flood.sh”. For example, you will probably like to deny further incoming packets that have been already classified as SPIT. In this case, you can define a firewall rule inside the script on_spit.sh which is called everytime that SPIT is detected. So, in this file you can configure the parameters to be sent to that script (such as IP addresses).

The list of available commands for the sections is defined in actions.conf itself. Here is a list of commands availables so far:

$From_Extension$
$From_IP$
$From_Port$
$From_Transport$
$Contact_IP$
$Contact_Port$
$Contact_Transport$
$Connection_IP$
$Owner_IP$
$Via_IP$ This is the first Via detected
$Via_Port$
$Tool_name$

For example, you can define the following configuration:

[actions]
on_flood=$Contact_IP$ $Contact_Port$ $From_Extension$ $Tool_name$
on_spit=$From_IP$ $From_Port$ $From_Extension$ $Owner_IP$ $Tool_name$
on_scanning=$Contact_IP$ $Contact_Port$ $Via_IP$ $Via_Port$ $From_Extension$ $Tool_name$

In this example, when a flood attack is detected and the on_flood.sh script is invoked, then all the commands will be replaced with the attacker data. For example:

bash on_flood.sh 10.0.30.2 5060 8888
